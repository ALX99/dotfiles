# Go 1.27 Agent Reference

Official source: <https://go.dev/doc/go1.27>

Go 1.27 is expected in August 2026. Until it is released, the official notes are a work in progress. Re-check them before implementing a feature, reviewing compatibility, or updating a module's `go` directive.

## Version Gate

1. Run `go list -m -f '{{.GoVersion}}'` and inspect any `toolchain` directive.
2. Use Go 1.27 language features and standard-library symbols only when the module intentionally targets `go 1.27` or later.
3. Do not raise the minimum version solely to make a local refactor shorter. Confirm the repository's supported Go versions and CI matrix first.
4. Expect `go test` to report standard-library symbols newer than the file's effective Go version through the default `stdversion` vet check.

## Language

### Generic concrete methods

Concrete methods may declare type parameters:

```go
type Stream[T any] struct {
    values []T
}

func (s Stream[T]) Map[U any](f func(T) U) Stream[U] {
    out := make([]U, len(s.values))
    for i, value := range s.values {
        out[i] = f(value)
    }
    return Stream[U]{values: out}
}
```

Use generic methods when the operation belongs to the receiver and method chaining improves readability. Keep using generic functions when they are simpler or more reusable.

Constraints:

- Interface methods cannot declare type parameters.
- A generic concrete method does not satisfy a non-generic interface method through one possible instantiation.
- Reflection cannot expose an uninstantiated generic method.
- Tooling that consumes Go syntax or `go/types` data may need updates; verify generators, analyzers, and code-indexing tools.

### Promoted fields in keyed struct literals

A promoted field from an embedded struct may be used directly as a key:

```go
type Retry struct {
    Attempts int
}

type Config struct {
    Retry
}

cfg := Config{Attempts: 3}
```

Use this only when promotion is unambiguous. Prefer `Config{Retry: Retry{Attempts: 3}}` when the explicit ownership is clearer or when compatibility with older Go versions matters.

### Broader generic function inference

A generic function assigned to a known function type can now infer its type arguments in all assignment contexts, including composite literals, conversions, and channel sends.

```go
func consume[T any](T) {}

callbacks := []func(string){consume}
```

Do not remove explicit type arguments when the target type is subtle or inference makes the code harder to understand.

## Tooling

### `go test`

- The `stdversion` vet analyzer now runs by default. A test command may fail because code refers to a standard-library API newer than the effective Go version.
- `go test -json` output events may contain `"OutputType"` for structured error/frame output. Parsers must ignore unknown fields and handle the new field when useful.

### `go doc`

- Use `go doc package@version` to inspect a specific module version.
- Use `go doc -ex` to list executable examples.
- Asking for a named example prints its source and comments.

### `go fix`

Go 1.27 adds these modernizers:

- `atomictypes`
- `embedlit`
- `slicesbackward`
- `unsafefuncs`

Treat `go fix` output like any automated refactor: inspect the diff, run focused tests, then run the repository's broader checks.

### `go mod tidy`

For modules declaring `go 1.27+`, `go mod tidy` consolidates duplicate `require` blocks into at most one direct and one indirect block while preserving associated comments. Do not hand-edit the file back into multiple blocks.

### Other command changes

- Compiler tools accept GCC-compatible `@response-file` arguments.
- `go tool trace -http=:6060` binds to localhost; specify `0.0.0.0:6060` only when remote access is intentional and safe.
- The `go` command no longer fetches modules from Bazaar (`bzr`) repositories.

## Runtime and Debugging

### Goroutine leak profile

The `goroutineleak` profile is generally available through `runtime/pprof` and `/debug/pprof/goroutineleak`.

Use it for blocked goroutines whose synchronization primitive is unreachable from runnable goroutines. It cannot prove the absence of every leak, especially when the primitive remains reachable through globals or runnable goroutine locals.

```go
profile := pprof.Lookup("goroutineleak")
if profile != nil {
    if err := profile.WriteTo(os.Stderr, 1); err != nil {
        return fmt.Errorf("write goroutine leak profile: %w", err)
    }
}
```

### Runtime behavior

- Small allocations below 80 bytes may be faster due to size-specialized allocation routines. Benchmark before changing code around this implementation detail.
- The binary may grow by roughly 60 KB. Do not disable the optimization unless a measured regression justifies `GOEXPERIMENT=nosizespecializedmalloc`.
- `asynctimerchan` is removed. Timer channels are always synchronous; delete compatibility branches that depend on restoring buffered timer-channel behavior.
- Tracebacks for `go 1.27+` modules include pprof goroutine labels unless `GODEBUG=tracebacklabels=0` is set. Never put secrets in labels.

## Standard Library

### `encoding/json/v2` and `encoding/json/jsontext`

Use `encoding/json/v2` for new Go 1.27+ code when you want configurable semantics and stricter interoperable defaults. It rejects invalid UTF-8 and duplicate object names by default.

```go
import json "encoding/json/v2"

err := json.Unmarshal(input, &dst, json.RejectUnknownMembers(true))
```

`encoding/json/jsontext` provides lower-level token/value streaming with validity enforcement. Prefer it for syntax-aware transforms; do not reimplement a JSON tokenizer.

Migration cautions:

- Review field-name matching, nil collection formatting, custom marshal methods, and duplicate-name behavior.
- The old `encoding/json` API remains supported and preserves behavior, but exact error text may change because it is backed by the v2 implementation.
- Tests should use typed errors or semantic assertions rather than complete error-string equality.

### `uuid`

The new standard `uuid` package supports generation, parsing, text encoding, comparison, and comparable `[16]byte` values.

```go
randomID := uuid.New()
orderedID := uuid.NewV7()
id, err := uuid.Parse(raw)
```

- Use `New()` when the algorithm does not matter.
- Use `NewV7()` for timestamp-ordered identifiers; ordering can be affected if the system clock moves backwards.
- Use `Parse`, not `MustParse`, for untrusted input.
- Replace a third-party UUID dependency only after checking database interfaces, JSON formats, UUID versions, and project-specific helpers.

### Cryptography

- `crypto/mldsa` implements FIPS 204 ML-DSA.
- `crypto/x509` and TLS 1.3 support ML-DSA keys and signatures.
- TLS adds ML-KEM-1024 support when explicitly configured.

Use these only for requirements that call for those algorithms. Do not invent protocol choices or replace established cryptographic configurations opportunistically.

### Experimental SIMD

`simd` and `simd/archsimd` require `GOEXPERIMENT=simd`; the API is not stable.

- Do not use them in ordinary application code by default.
- Require benchmarks, supported architecture checks, a scalar fallback, and explicit project approval.
- Prefer portable `simd` over architecture-specific `simd/archsimd` unless the architecture dependency is intentional.

### Useful APIs

Prefer these over hand-written equivalents when targeting Go 1.27+:

- `bytes.CutLast` and `strings.CutLast`
- `(*url.URL).Clone` and `url.Values.Clone`
- `database/sql.ConvertAssign` for driver-side `Rows.Scan` conversions
- `go/constant.StringLen`
- `go/scanner.Scanner.End`
- `math/big.Int.Divide` with explicit rounding mode
- `testing/synctest.Sleep`
- `net/http/httptest.NewTestServer` for synctest-compatible in-memory HTTP tests

## Compatibility Review Checklist

When reviewing an upgrade to Go 1.27, explicitly check:

- **JSON tests:** exact error strings and behavior around invalid UTF-8, duplicate names, or field matching.
- **Compressed fixtures:** `compress/flate` output may differ, affecting gzip, zip, zlib, PNG, and golden files even though decompressed content is equivalent.
- **Networking errors:** Unix socket reads now return `io.EOF` directly instead of wrapping it in `*net.OpError`.
- **HTTP response cleanup:** closing an HTTP/1 response body automatically drains a conservative amount for connection reuse. Still consume the body when application semantics require it.
- **HTTP/2 scheduling:** servers accept RFC 9218 client priority signals unless disabled.
- **Header limits:** consider `http.Server.MaxHeaderValueCount` for exposed servers.
- **Function pointers:** never compare function code pointers for equality; closure code sharing may make this fail in more cases.
- **macOS:** Go 1.27 requires macOS 13 Ventura or later.
- **Unicode:** tables move from Unicode 15 to Unicode 17; regenerate or review Unicode-sensitive fixtures.
- **Removed GODEBUG settings:** delete old fallbacks rather than trying to preserve values that Go 1.27 rejects.
